# 요구사항 정의

# 회원

## 회원가입

⭕️ Happy Path

- 사용자는 회원가입 화면에서 아이디, 이메일, 생년월일, 성별을 입력한다.
- 가입이 완료되면 서비스를 이용할 수 있다.

❌ Fail Path

- 이미 사용 중인 아이디로 가입하려 하면, "이미 사용 중인 아이디입니다" 안내가 표시된다.
- 필수 입력값(아이디, 이메일)을 누락하면 가입이 진행되지 않는다.

⚠️ Edge Case

- 아이디는 영문+숫자 10자 이내만 허용된다.
- 동시에 같은 아이디로 가입 요청이 들어오면, 먼저 처리된 요청만 성공한다.

## 내 정보 조회

⭕️ Happy Path

- 로그인한 사용자는 마이페이지에서 자신의 아이디, 이메일, 생년월일 등을 확인할 수 있다.

❌ Fail Path

- 로그인하지 않은 상태에서 접근하면 인증 오류가 발생한다.

# 상품

## 상품 목록

⭕️ Happy Path

- 사용자는 여러 상품을 한눈에 볼 수 있는 '상품 목록' 화면으로 이동한다.
- 화면에는 기본적으로 최신에 등록된 상품들이 먼저 보인다.
- 사용자는 '인기순', '가격순' 등 원하는 기준으로 상품을 다시 정렬할 수 있다.
- 특정 '브랜드'를 선택하여 그 브랜드의 상품만 모아볼 수 있다.
- 각 상품의 대표 이미지, 이름, 가격, 그리고 얼마나 많은 사람들이 '좋아요' 했는지 간략히 확인할 수 있다.

❌ Fail Path

- 판매 중인 상품이 하나도 없으면, "표시할 상품이 없습니다"와 같은 안내 문구가 표시된다.

⚠️ Edge Case

- 페이지네이션: 마지막 페이지를 넘는 페이지 번호를 요청하면 빈 목록이 반환된다.
- 존재하지 않는 브랜드 ID로 필터링하면 빈 목록이 반환된다.
- 좋아요 수는 LikeSummary 비정규화 데이터를 기반으로 조회하며, 실시간 정합성보다 조회 성능을 우선한다.

## 상품 상세

⭕️ Happy Path

- 사용자는 '상품 목록'에서 마음에 드는 상품을 선택한다.
- 선택한 상품만의 상세 페이지로 이동하며, 이곳에서 더 큰 이미지와 자세한 설명을 볼 수 있다.
- 해당 상품의 정확한 가격, 소속 브랜드 정보, 그리고 내가 이 상품을 '좋아요' 했는지 여부를 확인할 수 있다.

❌ Fail Path

- 판매가 중단되었거나 알 수 없는 상품 페이지에 접근하면, "해당 상품을 찾을 수 없습니다" 라는 안내가 표시된다.

⚠️ Edge Case

- 비로그인 사용자도 상품 상세를 볼 수 있지만, '좋아요' 여부는 표시되지 않는다.
- 브랜드 정보는 Redis 캐시에서 우선 조회하며, 캐시 미스 시 DB에서 조회한다.

# 브랜드

## 브랜드 목록 조회

⭕️ Happy Path

- 사용자는 전체 브랜드 목록을 조회할 수 있다.

## 브랜드 정보 조회

⭕️ Happy Path

- 사용자는 상품 상세 정보 등에서 브랜드 이름을 선택한다.
- 해당 브랜드의 전용 페이지로 이동하여, 브랜드 소개 글과 해당 브랜드의 모든 상품 목록을 볼 수 있다.

❌ Fail Path

- 존재하지 않는 브랜드 페이지에 접근하면, "해당 브랜드를 찾을 수 없습니다" 라는 안내가 표시된다.

⚠️ Edge Case

- 브랜드 데이터는 Redis 캐시를 통해 조회되며, 캐시 갱신 전까지 최신 변경이 반영되지 않을 수 있다.

# 좋아요

## 상품 '좋아요' 표시

⭕️ Happy Path

- 사용자는 상품 옆에 있는 '좋아요(하트)' 버튼을 누른다.
- 버튼이 활성화되며, 해당 상품이 나의 '좋아요 목록'에 저장된다.
- 이미 '좋아요'를 누른 상품의 버튼을 다시 누르면, '좋아요'가 취소되고 목록에서 제거된다.

❌ Fail Path

- 로그인을 하지 않은 상태에서 '좋아요' 버튼을 누르면, 로그인이 필요하다는 안내와 함께 로그인 화면으로 이동한다.

⚠️ Edge Case

- 같은 사용자가 같은 상품에 중복 좋아요를 누르면, 복합 유니크 제약(userId + productId)으로 중복이 방지된다.
- 좋아요/취소 시 LikeSummary(좋아요 수 집계)가 이벤트를 통해 비동기로 갱신된다.
- 존재하지 않는 상품에 좋아요를 누르면 오류가 발생한다.

## '좋아요'한 상품 목록 보기

⭕️ Happy Path

- 사용자는 '마이페이지' 등에서 '좋아요 목록' 화면으로 이동한다.
- 그동안 '좋아요'를 눌렀던 모든 상품들이 목록 형태로 나타난다.

❌ Fail Path

- '좋아요'를 누른 상품이 하나도 없으면, "좋아요 한 상품이 없습니다" 라는 안내 문구가 표시된다.

⚠️ Edge Case

- 좋아요한 상품이 이후 삭제/판매중단된 경우, 목록에는 표시되지만 상세 진입 시 안내가 표시된다.

# 장바구니

## 장바구니에 상품 담기

⭕️ Happy Path

- 사용자는 상품 상세 페이지에서 수량을 선택하고 '장바구니 담기' 버튼을 누른다.
- 장바구니에 상품이 추가되고, 헤더의 장바구니 뱃지 숫자가 갱신된다.

❌ Fail Path

- 로그인하지 않은 상태에서 장바구니 담기를 시도하면 인증 오류가 발생한다.

⚠️ Edge Case

- 이미 장바구니에 있는 상품을 다시 담으면 수량이 합산되어야 하는지, 별도 항목으로 추가되어야 하는지 정책 확인 필요.
- 장바구니에 담은 후 상품 가격이 변경되면, 주문 시점의 가격으로 계산해야 한다.

## 장바구니 조회

⭕️ Happy Path

- 사용자는 장바구니 페이지에서 담아둔 상품 목록, 각 상품의 수량과 가격을 확인한다.
- 전체 합계 금액을 확인할 수 있다.

❌ Fail Path

- 장바구니에 담긴 상품이 없으면, "장바구니가 비어있습니다" 안내가 표시된다.

## 장바구니 수량 변경

⭕️ Happy Path

- 사용자는 장바구니에서 특정 상품의 수량을 변경할 수 있다.
- 변경 후 합계 금액이 즉시 갱신된다.

❌ Fail Path

- 수량을 0 이하로 변경하려 하면 허용되지 않는다.

## 장바구니 상품 삭제

⭕️ Happy Path

- 사용자는 장바구니에서 특정 상품을 선택하여 삭제할 수 있다.
- '전체 비우기' 버튼으로 장바구니를 한번에 비울 수 있다.

# 포인트

## 포인트 조회

⭕️ Happy Path

- 사용자는 마이페이지에서 현재 보유한 포인트 잔액을 확인할 수 있다.

❌ Fail Path

- 로그인하지 않은 상태에서 접근하면 인증 오류가 발생한다.

## 포인트 충전

⭕️ Happy Path

- 사용자는 원하는 금액만큼 포인트를 충전할 수 있다.
- 충전 후 잔액이 즉시 반영된다.

❌ Fail Path

- 0 이하의 금액을 충전하려 하면 허용되지 않는다.

⚠️ Edge Case

- 동시에 여러 충전 요청이 들어오면, 각 요청이 순차적으로 처리되어 잔액 정합성이 보장되어야 한다.

# 주문

## 주문서 조회

⭕️ Happy Path

- 사용자가 '주문하기' 버튼을 누르면 주문서 페이지로 이동한다.
- 주문서에는 주문할 상품 목록, 수량, 가격, 배송지 입력란이 표시된다.
- 보유 포인트 잔액이 표시되며, 사용할 포인트를 입력할 수 있다.

❌ Fail Path

- 로그인하지 않은 상태라면 로그인 화면으로 안내된다.
- 장바구니가 비어있으면 주문서에 진입할 수 없다.

## 주문 생성

⭕️ Happy Path

- 사용자는 구매할 상품의 수량을 정하고 '주문하기' 버튼을 누른다.
- 배송 정보를 입력하고, 보유한 포인트를 사용하여 결제할 금액을 확인한다.
- 최종 '결제하기'를 완료하면 주문이 성공적으로 접수되고, "주문이 완료되었습니다" 라는 안내를 받는다.
- 주문한 만큼 상품의 재고와 사용자의 포인트가 차감된다.

❌ Fail Path

- 주문하려는 상품의 재고가 부족하면, "죄송합니다. 상품의 재고가 부족합니다" 라는 안내를 받는다.
- 결제에 필요한 포인트가 부족하면, "보유 포인트가 부족합니다" 라는 안내를 받는다.
- 로그인을 하지 않은 상태라면, 주문 전 로그인 화면으로 먼저 안내된다.

⚠️ Edge Case

- **동시 주문 시 재고 경합**: 여러 사용자가 동시에 같은 상품을 주문할 때, 재고 차감이 원자적으로 처리되어야 한다. 재고 부족 시 늦은 요청은 실패해야 한다.
- **주문번호 중복 방지**: 주문번호는 `yyyyMMddHHmmss` + 4자리 랜덤으로 생성되며, 동일 시각 충돌 가능성에 대비해야 한다.
- **주문번호 서명 검증**: 발급된 주문번호는 HMAC-SHA256 서명으로 위변조를 방지한다. 미발급/이미 사용된/서명 불일치 시 각각 다른 에러 코드를 반환한다.
- **금액 정합성**: 프론트에서 전달한 결제 금액과 서버가 재계산한 금액이 불일치하면 주문을 거부해야 한다.
- **포인트 차감 + 재고 차감 원자성**: 두 작업 중 하나라도 실패하면 전체 롤백되어야 한다.

## 유저의 주문 목록 조회

⭕️ Happy Path

- 사용자는 '마이페이지' 등에서 '주문 내역' 화면으로 이동한다.
- 과거부터 현재까지의 모든 주문 목록이 최신순으로 나타난다.
- 각 주문의 상태(예: 결제완료, 배송중, 배송완료)를 한눈에 확인할 수 있다.

❌ Fail Path

- 로그인을 하지 않은 상태라면, '주문 내역'을 보기 전 로그인 화면으로 먼저 안내된다.
- 지금까지 주문한 내역이 한 번도 없다면, "주문 내역이 없습니다" 와 같은 안내 문구가 표시된다.

## 주문 상세 조회

⭕️ Happy Path

- 사용자는 '주문 내역' 목록에서 특정 주문을 선택한다.
- 해당 주문의 상세 페이지로 이동하여, 어떤 상품을 몇 개 주문했는지, 총 결제 금액은 얼마인지, 배송지는 어디인지 등 자세한 정보를 모두 확인할 수 있다.
- 각 주문 상세 정보에는 주문 ID, 주문 상태, 주문 날짜, 총 금액, 상품 목록이 포함된다.

❌ Fail Path

- 다른 사람의 주문이거나 존재하지 않는 주문을 보려고 하면, "해당 주문 정보를 찾을 수 없습니다" 라는 안내가 표시된다.

# 결제

## 결제 준비 (Ready)

⭕️ Happy Path

- 사용자가 주문서에서 '결제하기'를 누르면, 서버에 결제 준비 정보(금액, 결제수단, 배송지)가 등록된다.
- Payment(PENDING), PaymentAttempt(REQUESTED) 레코드가 생성된다.

❌ Fail Path

- 주문번호가 유효하지 않으면(미발급, 이미 사용, 서명 불일치) 결제 준비가 거부된다.

⚠️ Edge Case

- **멱등성 보장**: 동일한 `orderNo + orderKey + READY` 조합으로 중복 요청이 오면, 캐시된 결과를 반환하고 새로운 레코드를 생성하지 않는다.
- **사용자 중복 클릭**: 결제 버튼을 빠르게 두 번 누르는 경우, 멱등키 덕분에 중복 결제가 방지된다.

## 결제 세션 생성 (Payment Session)

⭕️ Happy Path

- Ready 이후 PG사에 결제를 요청하여 결제 URL(paymentUrl)과 결제 키(paymentKey)를 확보한다.
- 사용자는 반환된 paymentUrl로 이동하여 PG 결제를 진행한다.

❌ Fail Path

- PG사 요청이 실패하면 PaymentEvent.Failure 이벤트가 발행되고, 결제가 진행되지 않는다.

⚠️ Edge Case

- **멱등성 보장**: `orderNo + orderKey + PAYMENT_SESSION` 조합으로 중복 방지.
- **PG 타임아웃**: PG사 응답이 지연되는 경우, 일정 시간 내에 응답이 없으면 실패 처리한다.

## 결제 완료 처리

⭕️ Happy Path

- PG 결제가 완료되면, 서버 채널(PG→서버 콜백)과 브라우저 채널(PG→브라우저→서버) 두 경로로 결제 결과가 전달된다.
- 서버는 PG 결제 결과를 검증하고, Payment 상태를 PAID로 갱신한다.
- PaymentEvent.Success 이벤트를 통해 Order 상태도 PAID로 전이된다.
- 사용자는 주문 완료 페이지로 이동한다.

❌ Fail Path

- PG 결제가 실패하면 Payment 상태가 FAILED로 갱신되고, 사용자에게 실패 안내가 표시된다.

⚠️ Edge Case

- **금액 위변조 방지**: PG가 전달한 금액과 서버에 저장된 결제 금액이 불일치하면 결제를 거부한다.
- **서버-브라우저 채널 순서**: 두 채널의 도착 순서가 보장되지 않으므로, 어느 쪽이 먼저 오더라도 정합성이 유지되어야 한다.

# 주문 취소

## 주문 취소 신청

⭕️ Happy Path

- 사용자는 주문 상세 페이지에서 '주문 취소' 버튼을 누른다.
- 취소 신청 페이지에서 환불 예정 금액을 확인한다.
- '취소 확인'을 누르면 주문이 취소되고 PG 환불이 진행된다.
- 사용한 포인트가 복원되고, 재고가 원복된다.
- Order 상태가 CANCELED로 변경된다.

❌ Fail Path

- 이미 배송이 시작된 주문은 취소할 수 없다.
- 이미 취소된 주문을 다시 취소하려 하면 오류가 발생한다.

⚠️ Edge Case

- **부분 취소**: 여러 상품 중 일부만 취소할 수 있다 (orderOptionNo 단위).
- **수량 단위 부분 취소**: 동일 상품의 수량 중 일부만 취소할 수 있다.
- **취소 중 상태 변경**: 사용자가 취소 화면에 머무르는 동안 다른 경로(CS, 시스템)로 주문 상태가 변경될 수 있으므로, 실행 직전 상태를 재확인한다.
- **쿠폰 복원**: 취소 시 사용한 쿠폰이 복원되어야 하며, 쿠폰 유효기간이 만료된 경우의 처리 정책이 필요하다.
- **PG 환불**: 카카오페이 등 PG 결제는 원래 결제수단으로 자동 환불된다. 무통장입금의 경우 환불 계좌 정보가 필요하다.

# 랭킹

## 일일/주간 랭킹 조회

⭕️ Happy Path

- 사용자는 랭킹 페이지에서 오늘의 인기 상품(일일 랭킹) 또는 이번 주 인기 상품(주간 랭킹)을 확인할 수 있다.

❌ Fail Path

- 랭킹 데이터가 없으면 빈 목록이 표시된다.

⚠️ Edge Case

- 랭킹은 Redis 기반으로 조회되며, 집계 주기에 따라 실시간 반영이 아닐 수 있다.

# 비기능 요구사항 (공통)

## 동시성

- 재고 차감, 포인트 차감, 좋아요 등 동시 요청이 가능한 자원에 대해 원자적 처리를 보장한다.
- 쿠폰 사용에는 낙관적 락(@Version)이 적용되어 있으며, 충돌 시 재시도 또는 실패 처리한다.

## 멱등성

- 결제 Ready, Payment Session 단계에서 IdempotencyKey를 통해 중복 요청을 방지한다.
- 동일 요청의 DataIntegrityViolationException은 정상 흐름으로 처리한다.

## 캐싱

- 브랜드, 상품 목록 등 읽기 빈도가 높은 데이터는 Redis 캐시를 적용한다.
- 캐시 미스 시 DB fallback 후 캐시를 갱신한다.

## 인증

- 인증이 필요한 API는 `User-ID` 헤더를 통해 사용자를 식별한다.
- 미인증 요청은 적절한 에러 코드를 반환한다.

## 이벤트 기반 처리

- 좋아요 수 집계, 조회수 기록 등 부가 처리는 Spring ApplicationEvent를 통해 비동기로 처리한다.
- 결제 성공/실패 시 이벤트를 통해 주문 상태를 전이한다.
